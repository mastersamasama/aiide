# 核心概念

这份文档解释 aiide 里那些反复出现的名词和分数，读完你能看懂 dashboard 上的每个字段、知道每个分数意味着什么、以及为什么 aiide 在很多地方宁可标 `n/a` 也不给你一个好看的数字。

不涉及具体操作——操作看 [快速上手](getting-started.md)、[观测指南](observability.md)、[评测指南](skill-lab.md)。

---

## 两条路径的区别

aiide 做两件事，心智模型不一样：

| | 观测（Observability） | 评测（Skill Lab） |
|---|---|---|
| 在做什么 | 对**已经跑完**的 session 记录做事后解剖 | 主动定义任务，让 agent 在**受控条件**下反复跑 |
| 数据从哪来 | 你导入的 `.jsonl` session 记录 | aiide 自己跑出来的实验 |
| 回答什么问题 | 「这段 session 到底发生了什么、为什么慢/贵」 | 「这个 skill 好不好用、哪一版更好」 |
| 需要什么 | 什么都不需要，零成本 | 一份 suite，以及能跑起来的被测 agent |

两条路径共享同一套底层数据模型（run、metrics），所以观测里学到的字段含义在评测里同样适用。

---

## 基本名词

- **run** —— 一段 session 被解析后的规范化结果。一个 `.jsonl` 文件对应一个 run，包含逐轮的回合、token 四桶、工具调用、skill 触发等。存在 `.aiide/runs/`。
- **experiment（实验）** —— 一次 `aiide lab run` 的完整产物：一份 suite × 一个模型 × N 次重复跑出来的所有 run，加上算好的分数和统计。**一旦封存不可变**——这是 aiide 保证「历史证据可信」的基础。存在 `.aiide/experiments/`。
- **skill** —— 在 aiide 里 skill 是一等公民。它会跨实验被追踪：不同版本、每次的触发率、占用多少 context token、分数趋势。
- **suite** —— 一份 JSON，定义一次评测：用哪个模型、装哪些 skill、跑哪些任务、每个任务怎么算对。允许写 `//` 和 `/* */` 注释。
- **verifier（验证器）** —— 判定一个任务答对没有的规则。四种类型：`regex`（答案匹配正则）、`numeric_range`（答案里出现落在区间内的数）、`json_field`（答案是 JSON 且某路径非空）、`file_exists`（工作区里生成了指定文件）。一个任务的所有 verifier 全过，这一次才算 C=1。
- **runtime** —— 被测的 agent 是什么。Claude Code 是**原生**的（直接导入、直接评测）；其它 agent（Codex、自建 CLI、HTTP 服务、网页产品）通过一个 **adapter（适配器）** 接入——一个小程序，收 prompt、调你的 agent、打印一个 JSON。观测不到的维度会诚实标 `n/a`，不因此扣分。见 [接入你自己的 agent](connect-your-agent.md)。
- **probe（探针）** —— 一份声明式 JSON 配置，告诉 aiide「某个外部 CLI / MCP 工具调用里，哪个是子命令」。有了 probe，覆盖率统计才能算出「这个工具的哪些命令被用到了」。放在 `.aiide/probes/`，没配也完全正常。

## 分数怎么读

评测的核心产物是 **SkillScore**——一个综合分，由四个维度加权而成：

| 维度 | 含义 | 权重 |
|---|---|---|
| **C** Correctness（正确性） | 任务的 verifier 是否全过。这是最硬的地面真相。 | 0.50 |
| **P** Process（流程） | 期望的 target skill 有没有被正确路由/触发。 | 0.25 |
| **R** Reliability（可靠性） | 多次重复之间结果是否一致。 | 0.15 |
| **H** Hygiene（卫生） | context 增长是否克制，有没有留出余量。 | 0.10 |

```
SkillScore = S 安全门 × 加权平均(C, P, R, H)
```

**关于 S（安全门）**：设计上 S 是一个乘法安全门——一旦触碰安全红线，综合分直接归零。当前 MVP 阶段 S 恒等于 1（即尚未启用实质的安全判定），所以现在它不会拉低分数，但公式里的位置已经预留好了。文档如实告诉你这一点，不假装它已经在把关。

**这些只做诊断，绝不进综合分：**

- **Efficiency（效率）**：耗时、成本、输出 token。它很有用，但它是诊断信号，不是质量——一个又快又便宜但答错的 skill 不该因为「效率高」而得高分。
- **pass@k**：k 次里至少对一次的概率，用来看稳定性。
- **activation×outcome**：触发了 skill 时的平均 C vs 没触发时的平均 C——用来看「触发这个 skill 到底有没有帮上忙」。

把效率和质量分开、只让确定性维度进综合分，是 aiide 的核心立场之一（deterministic-first）。

## 覆盖率：分数之外的另一半

分数回答「答对没、稳不稳」，但它回答不了另一个同样重要的问题：**这个 skill 是不是真在干活、值不值它占的那点 context？** 这由**覆盖率统计**（`aiide stats`）回答，它是判断「值不值得装」的关键：

- 一个 skill 装了、却**从没被触发**（有题考它却没触发）→ 纯粹的死重，白占 token。
- 一个 skill 随附的 reference 文件**从没被读**→ 可能可以删的累赘。
- 一个外部工具，实际用到的命令**只覆盖了它声明命令的一小部分**→ 剩下的可能是过度设计。

覆盖率的分母永远用「有效样本数」，跟分数用的样本不是一回事（细节见 [评测指南](skill-lab.md#32-覆盖率统计aiide-stats)）。这一整套统计同样恪守 null-not-zero：算不出来的就标 `n/a`，绝不用 0 冒充。

## 为什么分数只在「可比」时才连线

同一个分数，在不同的 suite、不同的模型、不同的 runtime 下跑出来，是不能直接比大小的——就像不同考卷的分数不能横向排名。aiide 把「同一份 suite（按字节哈希）· 同一个模型 · 同一个 runtime」称为一个**可比性 cohort（comparability cohort）**。

- skill 的分数趋势线，只在**同一个 cohort 内**连线；跨 cohort 的点单独列出，绝不连成一条误导性的曲线。
- 两个实验做 1v1 对比时，只有 cohort 完全一致才标为可因果比较；否则明确标注「仅相关，非因果」。

这条规则贯穿 CLI 和 dashboard，目的只有一个：不让你把噪声当成信号。

## 诚实披露：aiide 为什么常给你 `n/a`

aiide 有一套贯穿全代码的诚实鐵律，理解它们才能正确解读界面：

- **null 不是 0（null-not-zero）**：数据缺失就标 `n/a`，绝不用 0 填充。一个「没测到」的维度显示 0 会被误读成「得了零分」，那是在教你错误的结论。
- **env-noise 剔除，不算 C=0**：如果某次重复是因为环境噪声失败的（HTTP 429/529、连接被拒、认证过期等，命中重试白名单且重试耗尽），这一次会被**从分母中剔除**（标记 excluded），而不是当成一次答错。超时则仍算 C=0（那是真失败）。如果剔除后有效样本数掉到最低要求以下，综合分显示 `n/a` 并标注 degraded。
- **估算永远标注「估算」**：有些 context token 数无法精确得到（如工具结果、注入内容），aiide 会算但明确标为估算，绝不与精确值混为一谈。
- **不可见就不猜**：结构上观测不到的东西（如某些 runtime 的内部重试）宁可留空，也不编造。

## 治理原则

- **只读证据**：aiide 的所有产出都是证据，它从不回写你的 skill / suite，从不自动采用任何版本。**采用永远是人的决定。**
- **实验不可变**：封存后原始数据字节永不改动。dashboard 上给实验加注记也是写到独立 sidecar 文件，不碰原文件。
- **清理只走 CLI**：数据保留由 `aiide prune` 管理（默认只预览，`--yes` 才删）；dashboard 没有删除接口。

这些不是限制，而是让 aiide 的分数「敢信」的前提。
