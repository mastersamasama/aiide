# 為什麼要自建 Agent？——以實測數據說話的完整論證

> 基於真實 benchmark：同樣 3 道鏈上任務 × 3 次重複，「自建 crypto agent」對決「通用 agent（Claude Code）+ onchainos skills」。
> 所有數據可用 [aiide](../aiide/) 重現（`aiide lab run`），逐輪 trace 皆有存檔。2026-07。

---

## 執行摘要（一分鐘版）

兩邊**答對率打平（9/9 vs 9/9）**，但自建 agent：

- **快 3.4 倍**（15s vs 51s）
- **便宜 68 倍**（$0.004 vs $0.284/題；同模型換算後仍約 11 倍）
- **恆定 2 輪**完成（通用側 4–7 輪，不可預測）
- **工具零失誤**（通用側 10% 的調用被自家沙箱擋掉後繞路重試）

但「快和便宜」只是**症狀**。本文論證的是病因：通用 agent 有五種結構性的「給不了」，
它們無法靠寫更好的 skill、調更好的配置來彌補——因為那是 loop 所有權的問題，不是內容問題。

**一句話：skill 是把你的專業知識寄存在別人的大腦裡；自建 agent 是給你的專業知識配一顆自己的大腦。**

## 1. 實測證據

| 指標 | 自建 agent（DeepSeek-V4-Pro） | Claude Code + skills（sonnet） |
|---|---|---|
| 答對率 | 9/9 | 9/9 |
| 輪數（API 請求次數） | **恆定 2** | 4–7（均 5.4） |
| 總耗時 | 15.0s | 51.3s |
| 每題成本 | ~$0.004 | $0.284（API 實報） |
| 每輪 context | ~20.2k tokens，恆定 | 38–45k，遞增 |
| 工具失敗 | 0/18 | 3/31（全為沙箱摩擦） |
| 行為方差 | 零（每次同路徑） | 高（同題不同輪數） |

通用側 9 次執行的逐輪解剖，全部共享同一個固定骨架：

```
6/9 次：載入skill說明書 → 跑CLI → 讀說明書文檔 → 回答          （4 輪）
3/9 次：……中途撞沙箱（複合命令被擋/越界找檔案）→ 繞路重試 → …… （6–7 輪）
自建   ：調工具 → 回答                                          （2 輪，理論下限）
```

## 2. 差距解剖：三筆「通用性稅」

**① 間接層稅（固定 +2 輪，每次都付）。**
自建 agent 把工具以 typed schema 直接放進每次 API 請求——模型第一輪睜眼就能調用。
通用 agent 的 skill 是「說明書」：模型要先調 `Skill` 工具載入說明書、再讀說明書引用的文檔，才知道該跑什麼命令。
**工具印在考卷上 vs 開考才發課本**——這 2 輪與模型智力無關，是機制決定的。

**② 沙箱摩擦稅（隨機 +0~3 輪）。**
3 次工具失敗逐條驗屍：2 次是模型想把 CLI 回傳的 unix 時間戳轉成人類可讀時間、複合命令被權限系統擋下；
1 次是找 skill 參考文件時 `find /` 越出工作目錄被擋。**CLI 本身 9/9 一次成功、零失誤。**
通用沙箱為「不可信的任意任務」設計，跟窄領域任務摩擦生熱；自建 agent 把安全模型收窄到領域內（只暴露 8 個領域工具、全自動核准），這類輪整段消失。

**③ 基礎 context 稅（每輪 +20k tokens）。**
通用 agent 每輪背著完整通用 system prompt + 全套內建工具定義 + skill 清單（40.8k 起步）；
自建只帶領域 prompt + 8 個工具 schema（20.2k，可再壓）。乘上 2.7 倍輪數，累計讀取量差 5–8 倍。

**另有實測：熱進程差距。** 自建 agent 用長駐 MCP 子進程（啟動時握手一次，之後走熱 stdio 管道）；
通用側每次 Bash 都冷啟動 CLI——實測冷啟 2.5–7.2 秒/次（進程啟動 + 載入錢包 session + 新 TLS 連線）。

## 3. 病因：通用 agent 結構上給不了的五種控制權

以下每一條都對照了自建 runtime 的實際代碼，並經過三輪對抗性審查

### 3.1 大腦是「函數」，不是「常量」

自建 runtime 的 system prompt 是 `getSystemPrompt(locale, wallet, mode, memo, rules)`——
**每個用戶、每個 session 現場組裝一顆定製大腦**：錢包持倉直接寫進去（「你已經知道用戶持有什麼，不要問」）、
用戶的記憶、自訂規則、人格模式、語言偏好，全在第一輪 context 裡。

通用 agent 的 system prompt 是平台寫的常量，你只能「追加」——追加的內容和 4 萬 token 的通用指令競爭注意力；
用戶持倉這種數據只能讓模型多花一輪調工具去查。**「注入即答案」是 loop 所有者的特權。**

### 3.2 領域判斷力編進本能

自建 prompt 裡有一段錯誤碼決策樹：遇到 `50114/53017` 時，什麼情況是後端抖動該靜默重試、
什麼情況才真的要引導用戶登入、為什麼絕不能叫用戶去改憑證。這是踩過坑的運營智慧，被編碼成模型的本能。

Benchmark 裡通用 agent 遇到同一個錯誤碼時做了什麼？禮貌地告訴用戶「你的授權過期了，請重新登入」——然後答錯。
**同樣的錯誤碼，自建 agent 有 SOP，通用 agent 只有常識。**

### 3.3 反塑形：抹掉錯誤的本能

通用 coding agent 解 crypto 問題時滿腦子工程師動作——實測它去讀文檔、找檔案、寫 shell 換算時間。
自建 prompt 直接寫明「你在瀏覽器 app 裡，沒有終端，永遠不要建議用戶跑命令列」——**主動刪除錯誤的動作空間**。
通用 agent 不可能自宮自己的核心能力：它的思維鏈是為寫代碼塑形的，你的任務只是它眼裡的又一個工程問題。

> 這就是「harness 效應」的機制：同一顆底層模型，換個 harness 分數可差 23 個百分點、排名可逆轉。
> **模型決定智力上限，harness 決定職業身份。**

### 3.4 安全不變量用程式碼包圍

自建 runtime 裡，「swap 前必須向用戶確認」寫了三層：system prompt、工具 description、以及 dispatch handler 的程式碼。
加上輸入深度消毒、每對話預算熔斷（circuit breaker）、per-session 錢包隔離——**錢的動作被程式碼包圍，模型只是中間的建議者**。
通用 agent 的 hooks 能攔工具調用，但攔不住模型 prompt 裡本來就沒有的紀律。

### 3.5 Loop 物理層（連配置到極致的通用 agent 也做不到）

- **對話歷史手術**：剪掉已消化的工具結果、壓縮舊輪——通用 agent 的 transcript 只增不減，聊得越久越貴；自建可做到**任意對話長度下每題常數成本**
- **0-token fast path**：「ETH 多少錢」根本不必過模型——意圖命中 → 直接查 API → 模板回答，$0、<3s。頭部流量成本歸零
- **異構模型混搭**:便宜模型分類、DeepSeek 取數、強模型只做結論——通用 agent 鎖在單一模型家族，而且**連用戶用什麼模型都不是你決定的**
- **答案出門前的 gate**：數字驗證、合規檢查，不合格不出門
- **終局：蒸餾飛輪**——自己的流量 → 微調自有小模型 → 成本再降一個數量級。這是**唯一會隨時間變寬的護城河**；其他能力，通用 runtime 都可能明年內建

## 4. Skill 發佈者的視角：你交付的是產品，還是紙條？

以 skill 形式發佈領域能力，有六個你控制不了的變數：用戶用什麼**模型**、他環境裡有什麼**其他 skill/配置**在干擾、
他的**權限**設定會擋掉什麼、runtime **版本**何時漂移、你拿不到**遙測**（不知道用戶用得好不好）、
以及 skill **觸發本身是概率的**。

**託管 sandbox 可以救回大半**（統一版本、中心化更新、受控環境、可收遙測）——如果你的分發形態是「用戶在你的 sandbox 裡跑通用 runtime」，
skill 就從失控的紙條升級為受控的內容分發。但 sandbox 解決的是**分發與控制**，不解決 §2 的三筆稅和 §3 的五種控制權——那些是 runtime 的物理性質。

## 5. 不是二選一：skill 當內容層，runtime 按任務選

最優架構往往是混合的：

```
┌── 知識層：SKILL.md 倉庫（開放規格、跨 runtime 通用、中心化版本管理）──┐
│                                                                      │
├── 通用 runtime（sandbox 託管）              ├── 自建 agent
│    · 原生 skill 機制載入                     │    · 自建 loader：意圖路由 → skill body 直接注入
│    · 吃長尾/複合/沒鋪軌道的任務               │    · 吃 head 任務（查價、持倉、swap……）
│    · 通用思維鏈的應變力                       │    · 2 輪、fast path、答案 gate、常數成本
└───────────── 統一入口 + 混合路由：head → 自建；長尾 → 通用 ────────────┘
```

關鍵設計洞察：**同一份 SKILL.md，在通用 runtime 裡是「工具」（模型要花 2 輪去調用和閱讀），
在自建 agent 裡可以是「路由目標」（router 命中意圖後把 skill body 直接拼進第一輪 context——間接稅 = 0）。**
一份知識、兩個引擎，寫一次兩邊用。

## 6. 誠實面板:什麼時候不該自建

- **探索期**：寫 SKILL.md 一天，自建產品一個月——需求沒驗證前別蓋工廠
- **用戶住在通用 runtime 裡**（開發者工具場景）：skill 是零摩擦觸達
- **長尾為主的任務分佈**：rails 覆蓋不了的怪問題，通用思維鏈的應變力反而是資產

## 7. 決策清單

| 你的情況 | 建議 |
|---|---|
| 驗證想法、流量小、任務雜 | 通用 runtime + skill |
| 工具不可靠、輪數失控 | 先把工具做成 typed MCP server——最便宜的大改善 |
| head 流量集中、延遲/成本敏感、要 SLA | 自建 loop（本文全部論證生效） |
| 要金融級安全不變量、答案 gate、審計 | 自建 loop + 程式碼包圍 |
| 百萬級日活、要成本護城河 | 自建 + 蒸餾自有小模型 |

**判別式：一個優化需要控制什麼？工具內（任何形式都能做）→ runtime 配置（通用可調）→ loop 控制（只有自建）→ 推論權重（只有自有模型）。每往下一層，能力越稀有，維護債也越重——讓流量形狀決定你停在哪層。**
